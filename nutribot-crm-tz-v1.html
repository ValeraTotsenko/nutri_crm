01 — Обзор проекта

Что строим и зачем

Telegram Mini App + бот для ведения практики нутрициолога. Система заменяет ассистента в рутинных задачах: учёт клиентов, контроль оплат, управление задачами, воронка лидов и умные уведомления.

project.config
// Стек технологий
{
  platform: "Telegram Mini App",
  frontend: "React + Vite",
  backend: "Node.js (Fastify)",
  database: "PostgreSQL",
  scheduler: "node-cron / BullMQ",
  bot: "node-telegram-bot-api",
  deploy: "Railway / Render",
  timezone: "Europe/Kyiv",
  notifications_time: "10:00",
  notifications_days: "Mon–Fri"
}
Пользователь системы: один (нутрициолог-владелец). Никакого публичного доступа — всё закрыто на Telegram ID владельца.
Уведомления: только владельцу в личку бота. Пн–Пт в 10:00. Срочные события (день рождения, созвон завтра) — вне зависимости от расписания.
Расширяемость: структура БД и архитектура спроектированы с запасом — поля, типы уведомлений и логика добавляются без переделки ядра.

04 — Типы работы с клиентом

6 форматов — разная логика

Тип работы определяет: нужны ли созвоны, сколько их, на какой срок рассчитана работа и какие уведомления генерировать.

Тип работы Срок Созвоны Продление Особенности
individual
Индивидуальное сопровождение		 3 мес, продлевается Вводный + конец 1-го мес + 1–2/мес далее +1, +2 мес и т.д. Основной продукт. Полный цикл уведомлений
family
Семейное сопровождение		 3 мес, продлевается Аналогично индивидуальному +1, +2 мес и т.д. Несколько членов семьи, одна карточка клиента
pregnancy_planning
Планирование беременности		 3 мес, продлевается Аналогично сопровождению Да Особые противопоказания в карточке
pregnancy_support
Сопровождение беременности		 3 мес, продлевается Аналогично сопровождению Да Срок беременности в заметках
express
Экспресс		 Фиксированный (1 мес) 1 созвон (первый) → схема на 3 мес После завершения может перейти на схему_3m без созвонов
scheme_3m
Повторная схема на 3 мес		 3 мес Иногда 1 созвон (не всегда) Ещё одна схема Для тех, кто прошёл 3+ мес сопровождения или экспресс. Флаг has_call в карточке
Логика «Повторной схемы»: Схема на 3 мес без сопровождения доступна только после завершения минимум 3 мес сопровождения (individual / family / pregnancy_*) ИЛИ после экспресса. В карточке клиента — поле has_call: boolean (нужен ли созвон для конкретного клиента на схеме).
05 — Логика созвонов

Когда и какой созвон

Созвоны хранятся в таблице calls. Каждый созвон — отдельная запись с датой, типом и статусом. Уведомление — за 1 день до созвона.

Типы созвонов (call_type)
call_types.enum
intro // Вводный — первый созвон. Всегда для сопровождений.
// Для экспресса — тоже первый и единственный.
// Для схемы_3м — только если has_call = true

monthly // Конец 1-го мес сопровождения. Плановый.
// Далее — 1–2 в месяц по договорённости.

extra // Внеплановый созвон (по необходимости)

free_diagnostic // Бесплатная диагностика лида (30 мин).
// Привязывается к leads, не к clients.
// После — лид меняет статус.
График созвонов на сопровождении (3 мес)
День 1 — Старт
Вводный созвон (intro)
Знакомство, сбор анамнеза, постановка целей. Создаётся при добавлении клиента.
call_type: intro~60 мин
~30 день — Конец 1-го месяца
Плановый созвон (monthly)
Разбор первого месяца работы, корректировка плана. Нутрициолог создаёт вручную.
call_type: monthly~60 мин
Месяц 2–3 — По договорённости
1–2 созвона в месяц
Создаются по мере договорённости с клиентом. Тип: monthly или extra.
call_type: monthly / extra
После 3+ мес — Опционально
Продление или переход на схему
Продление сопровождения (+1 мес) или переход на схему_3м без созвонов (если has_call = false).
status: extendedwork_type → scheme_3m
Важно: Нутрициолог создаёт созвоны вручную в Mini App (дата + время + тип). Система не генерирует их автоматически — только напоминает о запланированных. Уведомление приходит за 1 день в 10:00, если созвон на следующий день.

🎯
Бесплатная диагностика
Лид записывается на БД — создаётся запись в calls с типом free_diagnostic. Флаг had_free_diagnostic = true. Уведомление за 1 день до созвона.
После БД нутрициолог вручную меняет статус лида
✅
Вошёл в работу
После БД лид заключил договор → кнопка «Перевести в клиента». Создаётся запись в clients, source_lead_id ставится, лид закрывается. Вся история сохраняется.
lead.converted_to_client_id = clients.id
💜
Купил закрытый клуб
Не вошёл в активную работу, но купил доступ в клуб. Статус → club_member. Остаётся в воронке как тёплый, напоминания продолжаются с учётом интереса к клубу.
Периодические напоминания о связи сохраняются
❌
Отказался / не отвечает
После БД пропал — не отвечает. Статус → refused. Уведомления прекращаются. Запись сохраняется в истории.
Уведомления по refused не отправляются
🔔
Напоминание о контакте
Для статусов new, warm, negotiations, club_member: если прошло remind_after_days с last_contact_at — напоминание. Default: 7 дней.
Каждое напоминание содержит кнопку «Написала» — обновляет last_contact_at
⚠️
Давно без контакта
Автоматический триггер: если last_contact_at старше 14 дней и статус не refused — лид помечается «требует внимания» и попадает в дайджест.

07 — Система уведомлений

11 триггеров — полная логика

Все уведомления идут в личку владельца в Telegram. Расписание: Пн–Пт, 10:00 (Europe/Kyiv). Каждое сообщение содержит кнопки быстрых действий.

Планировщик (Scheduler): Запускается ежедневно в 10:00 Пн–Пт. Проверяет все условия ниже и отправляет накопленные уведомления. Порядок отправки: сначала ДР → созвоны → дедлайны задач → оплаты → лиды → дайджест. Если сегодня суббота или воскресенье — не запускается.
